# מערכת לעיבוד וניתוח נתוני צי רחפנים (Drone Fleet Data Processing)

פרויקט זה פותח כחלק מדרישות מערכת עבור יחידת רחפנים צבאית. המערכת קולטת נתוני רחפנים מקובץ גולמי (JSON), מבצעת תהליך של בקרת איכות וולידציה לפי תקנים מחמירים, שומרת את הנתונים התקינים, ומפיקה דוח סטטיסטי מסכם אודות מצב הצי.

## 1. ניתוח המערכת

**תיאור המערכת והתהליכים המרכזיים:**
המערכת משמשת כצינור עיבוד נתונים (Data Pipeline) חד-פעמי למצב הצי היומי. התהליך המרכזי מתחלק למספר שלבים ברורים:
1. **קריאה (Extract):** טעינת נתוני JSON מקובץ גולמי.
2. **סינון ואימות (Validate):** בדיקת כל רשומה מול חוקים עסקיים מחמירים (טווחים, פורמט, היגיון תפעולי).
3. **שמירה (Load):** כתיבת הרשומות התקינות בלבד לקובץ נקי (`drones_clean.json`).
4. **ניתוח (Analyze):** טעינה מחדש של הנתונים הנקיים וביצוע שאילתות סטטיסטיות.
5. **דיווח (Report):** הפקת קובץ טקסט מעוצב (`analysis_report.txt`) המציג את תמונת המצב של הצי.

**האתגרים העיקריים:**
- התמודדות עם נתונים שגויים או חלקיים מבלי להקריס את המערכת.
- הפרדה מוחלטת בין תהליכי הקלט/פלט לבין הלוגיקה העסקית.
- עמידה בכללי SOLID כדי לאפשר בעתיד החלפה קלה של מקור הנתונים (למשל, מעבר מקובץ JSON למסד נתונים או API).

## 2. ארכיטקטורה

המערכת בנויה בארכיטקטורה שכבתית (Layered Architecture), עם הפרדת תחומי אחריות (Separation of Concerns).

**רכיבים ואחריות:**
- **שכבת המודלים (Models):** הגדרת ישויות המידע (`Drone`, `FleetReport`) ו-Enums. 
- **שכבת הקלט/פלט (I/O & Services):** מחלקות האחראיות על תקשורת עם מערכת הקבצים (`DroneDataLoader`, `DroneDataSaver`, `DroneDataReporter`, `FileWriter`).
- **שכבת הוולידציה (Validators):** מחלקות לבדיקת תקינות המידע, מבוססות ממשק (`IDroneValidator`) כדי לאפשר גמישות בשינוי חוקי התקינות.
- **שכבת העיבוד (Analytics):** מחלקת `DroneAnalyzer` המבצעת אגרגציות ושאילתות LINQ ליצירת דוח הנתונים.

**תקשורת בין הרכיבים:**
ה-`Program.cs` משמש כבקר (Controller) המנהל את זרימת המידע. הוא מקבל את המידע מה-Loader, מעביר אותו ל-Validator, שולח את התקינים ל-Saver, קורא ל-Analyzer ליצירת אובייקט הדו"ח, ולבסוף מעביר את הדו"ח ל-Reporter להדפסה. רכיבי הלוגיקה לעולם אינם ניגשים ישירות לקבצים.

## 4. רשימת רכיבים

| רכיב | תפקיד | מידע שהוא מחזיק | פעולות שהוא מספק | קשרים לרכיבים אחרים |
|---|---|---|---|---|
| `Drone` | מודל נתונים המייצג רחפן | מזהה, דגם, סטטוס, סוללה וכו' | `ToSummary()` | משמש את כל שכבות המערכת |

## 5. החלטות תכנוניות (Design Decisions)

1. **עקרון האחריות היחידה (SRP):**
   הפרדנו לחלוטין בין אובייקט שמבצע שאילתות (`DroneAnalyzer`) לבין אובייקט שמדפיס את התוצאות לקובץ (`DroneDataReporter`). ה-Analyzer מחזיר אובייקט נתונים נקי (`FleetReport`), מה שמאפשר בעתיד לשלוח את הדו"ח ל-API או ל-GUI מבלי לשנות את לוגיקת החישוב.
2. **תלות בהפשטות (Dependency Inversion - DIP):**
   המחלקה `DronesDataValidator` לא תלויה ישירות ב-`DroneValidator`, אלא בממשק `IDroneValidator`. כך, אם בעתיד יהיה צורך בחוקי ולידציה חדשים (למשל, עבור מודל רחפן חדיש), נוכל להזריק ולידטור אחר מבלי לשנות את לוגיקת הסינון.
3. **התמודדות עם שגיאות ונתיבים קשיחים:**
   לא השתמשנו בנתיבים מוחלטים (כמו `C:\Users\...`). במקום זאת, השתמשנו ב-`AppContext.BaseDirectory` בשילוב עם `Path.Combine` ו-`Path.GetFullPath`. המערכת מחפשת את התיקיות באופן יחסי לקובץ הריצה, כך שהפרויקט יעבוד בכל מחשב או שרת אליו יועבר. בנוסף, חריגות מקוסטמות (Custom Exceptions) הוגדרו כדי לתפוס בדיוק בעיות I/O (כגון קובץ חסר או JSON פגום) ולדווח עליהן בצורה מסודרת ל-Console מבלי להקריס את האפליקציה.
4. **שימוש נרחב ב-LINQ:**
   החלטנו להשתמש ב-LINQ במחלקת ה-`DroneAnalyzer` לטובת קריאות וקיבוץ נתונים (Grouping, Ordering, Aggregation) במקום בלולאות מסורתיות. זה הופך את הקוד לקריא, קצר וקל לתחזוקה. הוספנו ניתוח ספציפי של "בסיסים עם רחפנים מבצעיים בעלי סוללה חזקה".

## 6. תכנון חלוקת העבודה (Team Workflow & Git)

העבודה בפרויקט חולקה בין שני חברי הצוות על בסיס תחומי אחריות כדי למנוע התנגשויות (Merge Conflicts) ולאפשר פיתוח מקבילי (Parallel Development).

**חלוקת תפקידים:**
- **אלעזר גרשי:**
  אחראי על פיתוח תשתיות הקלט/פלט (I/O) של המערכת. התפקיד כולל יצירת התקשורת מול מערכת הקבצים, ניהול השגיאות החיצוניות ועיצוב התוצרים הסופיים.
  - *מימוש מחלקות:* `DroneDataLoader` (קליטת הנתונים), `DroneDataSaver` (שמירת האובייקטים התקינים), `DroneDataReporter` (כתיבת הדוח המסכם), ו-`FileWriter`.

- **אשר פירוב:**
  אחראי על הלוגיקה העסקית, הולידציה והניתוח האנליטי. התפקיד כולל את ייצוג המידע במערכת, סינון הנתונים לפי חוקי היחידה המחמירים, והפקת המידע הסטטיסטי באמצעות LINQ.
  - *מימוש מחלקות:* אובייקט ה-`Drone`, אובייקט התוצאות `FleetReport`, הגדרות ה-`Enums`, מחלקות הוולידציה `DroneValidator` ו-`DronesDataValidator`, ומחלקת הניתוח `DroneAnalyzer`.

**אסטרטגיית Git ותיאום:**
- ענף ה-`main` נשמר תמיד יציב (Stable).
- כל אחד מחברי הצוות פתח Feature Branch ייעודי לחלקים עליהם הוא עבד (למשל: `feature/io-handlers`, `feature/business-logic`).
- בסיום מימוש של רכיב משמעותי, נפתח Pull Request. בוצע Code Review הדדי כדי לוודא עמידה בדרישות ובתקני SOLID, ורק לאחר מכן בוצע Merge לענף הראשי.
- הממשקים תוכננו מראש בתחילת העבודה כך שאלעזר יוכל לקרוא ולשמור נתונים תוך שימוש במודלים שאשר הגדיר, ואשר יכול היה להעביר נתונים ל-Reporter שאלעזר כתב מבלי להמתין לסיום המימוש של הצד השני.
